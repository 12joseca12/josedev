# P4 ‚Äî Technical SEO Audit: josecoded.com (pre-deploy gate)

Date: 2026-07-20 ¬∑ Scope: read-only static analysis of `src/app/**`, `next.config.ts`, `public/`, literals.
Stack: Next.js 16.2.4 App Router, bilingual `/es` (default) + `/en`, no middleware, no root page.

Verdict: **NOT ready to deploy from an SEO standpoint.** 6 High, 8 Medium, 6 Low findings. The two worst issues are structural: the apex URL `/` returns 404, and `metadataBase` is effectively unset so every canonical/OG URL outside `/blog` resolves against `localhost`.

---

## HIGH severity

### H1. The root URL `/` returns 404 ‚Äî the domain apex is a dead page
- **Evidence:** `src/app/` contains only `[locale]/`, `demo/`, `stack/` (both empty), `favicon.ico`, `globals.css`, `sitemap.ts`. There is **no `src/app/page.tsx`**, **no `middleware.ts` anywhere in the repo** (verified with `find`), and `next.config.ts:4-9` only redirects `/stack`. A required dynamic segment `[locale]` cannot match the empty path.
- **Impact:** `https://josecoded.com/` ‚Äî the URL people type, link to, and that accumulates backlinks ‚Äî serves Next's default 404. Google will index nothing at the apex; all external links to the bare domain are wasted. This alone can sink the site.
- **Fix (pick one):**
  1. Add `src/middleware.ts` that 308-redirects `/` ‚Üí `/es` (and optionally Accept-Language-negotiates to `/en`), or
  2. Add a redirect in `next.config.ts`: `{ source: "/", destination: "/es", permanent: false }` (use `permanent: false`/307 if you may later language-negotiate; 308 if `/es` is forever the apex target).
  Then add `x-default` hreflang pointing at `/es` (see H3).

### H2. `metadataBase` missing at root; `NEXT_PUBLIC_SITE_URL` not set in env
- **Evidence:** `metadataBase` exists **only** in `src/app/[locale]/(site)/blog/layout.tsx:3-8`. The root locale layout `src/app/[locale]/layout.tsx:19-40` sets relative `alternates.canonical`/`languages` with no `metadataBase`. `.env.example:18` still has the placeholder `NEXT_PUBLIC_SITE_URL=https://tudominio.com`, and `.env.local` does not define it at all. `src/app/sitemap.ts:8` falls back to `http://localhost:3000`.
- **Impact:** Outside `/blog`, Next resolves relative canonical/hreflang/OG URLs against its fallback (localhost / deployment host on Workers ‚Äî there is no `VERCEL_URL`). You'd ship `<link rel="canonical" href="http://localhost:3000/es">` on the homepage and a sitemap full of `http://localhost:3000/...` URLs. Search engines will ignore or misattribute everything.
- **Fix:** In `src/app/[locale]/layout.tsx` `generateMetadata`, add:
  ```ts
  metadataBase: new URL(process.env.NEXT_PUBLIC_SITE_URL ?? "https://josecoded.com"),
  ```
  Delete the now-redundant `blog/layout.tsx` metadataBase (child layouts inherit). **Set `NEXT_PUBLIC_SITE_URL=https://josecoded.com` in the production Workers env ‚Äî make this a deploy checklist item.** Prefer a hard-coded production fallback over `localhost`.

### H3. Root-layout canonical/hreflang is inherited by every child page ‚Üí `/foro*` canonicalizes to the homepage
- **Evidence:** `src/app/[locale]/layout.tsx:25-28` sets `alternates: { canonical: "/${locale}", languages: { es: "/es", en: "/en" } }`. Next.js metadata **merges by field from layout to page**: any page that doesn't define its own `alternates` inherits the layout's whole `alternates` object. Pages with no `alternates` of their own: `foro/page.tsx`, `foro/[thematicSlug]/page.tsx:23-30`, `foro/[thematicSlug]/[entrySlug]/page.tsx:17-30`, `foro/new/page.tsx`, all `admin/*`, `staff/*`, `closer/*`, `area-clientes/*`, `auth/page.tsx`.
- **Impact:** Every forum page (public, in the sitemap at `sitemap.ts:17`) emits `<link rel="canonical" href=".../es">` ‚Äî telling Google "this page is a duplicate of the homepage." Google will drop `/foro`, every thematic, and every thread from the index. The inherited `languages: {es:/es, en:/en}` also puts homepage-pointing hreflang on blog articles (`blog/[slug]/page.tsx:45` sets `canonical` but not `languages`, so languages is inherited).
- **Fix:**
  1. Remove `alternates` from the root layout entirely; canonicals belong on pages.
  2. Home canonical moves to a tiny `generateMetadata` in `(site)/page.tsx` (or keep in layout but ensure every indexable page overrides ‚Äî fragile, not recommended).
  3. Add per-page `alternates` to `foro/page.tsx`, `foro/[thematicSlug]/page.tsx`, and `foro/[thematicSlug]/[entrySlug]/page.tsx` (`canonical: /${locale}/foro/...`, `languages` incl. both locales + `x-default`).
  4. In `blog/[slug]/page.tsx`, either add correct per-article `languages` (only if a translated slug actually exists) or explicitly set `alternates: { canonical, languages: undefined }` once the layout inheritance is removed.
  5. Everywhere `languages` is emitted, add `"x-default": "/es"`-style entries (Next: key `"x-default"` is supported in `alternates.languages`). Currently **no page emits x-default** (checked home, sobre-mi, services, blog index).

### H4. No `robots.ts` / `robots.txt` at all
- **Evidence:** `find src/app -name "robots*"` ‚Üí nothing; `public/` contains only the default Next SVGs (`file.svg`, `globe.svg`, `next.svg`, `vercel.svg`, `window.svg`).
- **Impact:** No sitemap declaration for crawlers, no crawl-path hygiene for `/admin`, `/area-clientes`, `/closer`, `/staff`, `/auth`, `/foro/new`, `/perfil`. Meta-noindex mostly covers indexing (see H5 for the gap), but crawl budget is wasted and the sitemap is undiscoverable except via Search Console.
- **Fix:** Add `src/app/robots.ts`:
  ```ts
  import type { MetadataRoute } from "next";
  const base = process.env.NEXT_PUBLIC_SITE_URL ?? "https://josecoded.com";
  export default function robots(): MetadataRoute.Robots {
    return {
      rules: [{
        userAgent: "*",
        allow: "/",
        disallow: ["/es/admin", "/en/admin", "/es/area-clientes", "/en/area-clientes",
                   "/es/closer", "/en/closer", "/es/staff", "/en/staff",
                   "/es/auth", "/en/auth", "/es/perfil", "/en/perfil", "/es/foro/new", "/en/foro/new"],
      }],
      sitemap: `${base}/sitemap.xml`,
    };
  }
  ```
  (Keep meta noindex on those pages too ‚Äî belt and braces; robots-disallowed pages can still be indexed URL-only if linked, but these are behind auth.)

### H5. `admin/page.tsx` (dashboard root) and `foro/new/page.tsx` have NO metadata ‚Üí they inherit `robots: { index: true }` from the root layout
- **Evidence:** `grep -L "generateMetadata|export const metadata"` hits `src/app/[locale]/admin/page.tsx` and `src/app/[locale]/foro/new/page.tsx`. Neither `admin/layout.tsx` nor `area-clientes/layout.tsx` nor `closer/layout.tsx` define `robots`. Meanwhile `src/app/[locale]/layout.tsx:24` sets `robots: { index: true, follow: true }` ‚Äî which those pages inherit. Every *other* admin/staff/area-clientes/closer/auth page correctly sets `robots: { index: false, follow: false }` (verified: `auth/page.tsx:19`, `admin/leads|finanzas|packs|chats|clientes|clientes/[id]`, `staff/login:13`, `staff/onboarding:12`, all 5 `area-clientes/*`, `closer/page.tsx:13`, `perfil:13`).
- **Impact:** `/es/admin` (the admin dashboard) and `/es/foro/new` (post-creation form) are explicitly marked indexable. This is the "1 page missing generateMetadata" ‚Äî it's the worst possible one to miss.
- **Fix:** Best structural fix: put `robots: { index: false, follow: false }` in `admin/layout.tsx`, `area-clientes/layout.tsx`, `closer/layout.tsx` (and a `staff/layout.tsx`) via `export const metadata` so no future page in those trees can be indexable by omission. Add a minimal `generateMetadata` (title + noindex) to `admin/page.tsx` and `foro/new/page.tsx`. Also remove `robots: { index: true ... }` from the root layout (indexable is the default; declaring it there is what created this trap).

### H6. No OG image anywhere ‚Äî `summary_large_image` Twitter card with no image
- **Evidence:** No `openGraph.images` in any page/layout (grep across `src/app`); no `opengraph-image.*` file convention anywhere; `public/` has no brand asset. Yet `[locale]/layout.tsx:34-38` and `blog/[slug]/page.tsx:53-57` declare `twitter: { card: "summary_large_image" }`.
- **Impact:** Every share on X/LinkedIn/WhatsApp/Slack renders text-only or a scraped random image. For a portfolio whose distribution is largely social/DM link-shares, this is a major CTR loss. `og:image` must also be an **absolute** URL ‚Äî which additionally depends on H2.
- **Fix:** Add a 1200x630 default image at `src/app/[locale]/opengraph-image.png` (Next auto-generates the absolute `og:image` + sizes for the whole tree), or set `openGraph: { images: [{ url: "/og/default.png", width: 1200, height: 630 }] }` in the root layout once `metadataBase` exists. For blog articles, consider dynamic OG via `opengraph-image.tsx` (ImageResponse) keyed on the post title ‚Äî high leverage for a technical blog.

## MEDIUM severity

### M1. No title template ‚Üí bare, unbranded, duplicate-prone titles
- **Evidence:** `[locale]/layout.tsx:22` sets `title: t(locale, "app.metadata.title")` as a plain string, not `{ default, template }`. Actual values (from `src/lib/literals.json` / `literalsEn.json`): es home = "JoseCoded" (9 chars); en home = "Full Stack ¬∑ Mobile ¬∑ Architecture" (no brand!); blog index title = "Blog" (4 chars, the entire tag); foro = "Foro"/"Forum"; services = "Servicios para negocios" (no brand).
- **Impact:** "Blog" as a full title is near-worthless in SERPs; en/es brand inconsistency splits brand-query signals; forum threads hand-roll branding (`entrySlug/page.tsx:27` does `` `${entryTitle} ¬∑ ${titleBase}` `` manually).
- **Fix:** In root layout: `title: { default: t(locale, "app.metadata.title"), template: "%s ¬∑ JoseCoded" }`. Unify the brand: es and en home titles should both contain "JoseCoded" (e.g. "JoseCoded ‚Äî Ingenier√≠a web y m√≥vil" / "JoseCoded ‚Äî Full-stack web & mobile engineering", 45-60 chars). Then per-page titles drop their hand-rolled "¬∑ Jose Dev" suffixes ("Jose Dev" vs "JoseCoded" is also inconsistent ‚Äî pick one brand string).

### M2. `/stack` redirect lands on a 404
- **Evidence:** `next.config.ts:6` ‚Äî `{ source: "/stack", destination: "/sobre-mi", permanent: true }`. But `/sobre-mi` without a locale prefix does not exist (no middleware, no root-level route). The locale-scoped rule on line 7 is correct.
- **Impact:** Legacy `/stack` backlinks 301 into a 404 ‚Äî link equity destroyed while looking like it's preserved.
- **Fix:** `{ source: "/stack", destination: "/es/sobre-mi", permanent: true }` (or to `/:locale/sobre-mi` via the H1 middleware after locale detection). Also delete the empty leftover dirs `src/app/stack/` and `src/app/demo/`.

### M3. Unsupported locales soft-render as Spanish (unbounded duplicate URLs)
- **Evidence:** `src/services/literals.ts:48-50` ‚Äî `resolveLocaleParam` falls back to the default locale instead of 404ing. `generateStaticParams` in `[locale]/layout.tsx:15-17` only emits es/en but `dynamicParams` is not set to `false`, so `/fr`, `/de/blog`, `/anything` render on demand as Spanish pages with HTTP 200.
- **Impact:** Infinite crawlable duplicate URL space. Partially self-healed because the inherited canonical resolves to `/es` ‚Äî but that same inheritance is what H3 removes, and soft-200s waste crawl budget regardless.
- **Fix:** In `[locale]/layout.tsx`, add `export const dynamicParams = false;` (Next then 404s unknown locales for statically-declared params), or call `notFound()` when `!isSupportedLocale(locale)` in the layout. Keep `resolveLocaleParam` for internal defensive use only.

### M4. Blog `Article` JSON-LD is incomplete
- **Evidence:** `blog/[slug]/page.tsx:79-91` ‚Äî has `headline`, `inLanguage`, `datePublished`, `dateModified`, `keywords`, `author.name` ("Jose Dev"). Missing: `image` (required for Article rich results), `url` / `mainEntityOfPage`, `description`, `publisher`, `author.url`.
- **Impact:** Ineligible for article rich results; weaker entity linking to the author.
- **Fix:** Add to the object:
  ```ts
  description, image: [absoluteOgImageUrl],
  mainEntityOfPage: `${base}/${locale}/blog/${post.slug}`,
  author: { "@type": "Person", name: "Jose ...", url: `${base}/${locale}/sobre-mi` },
  publisher: { "@type": "Person", name: "JoseCoded", url: base },
  ```
  Consider `@type: "BlogPosting"` (more specific, same eligibility).

### M5. Missing site-wide `Person`/`ProfessionalService` + `WebSite` JSON-LD
- **Evidence:** Only two JSON-LD blocks exist in the whole app: `FAQPage` on home (`(site)/page.tsx:15-33`) and `Article` on blog posts. No `Person`, no `WebSite`, no `Organization`/`ProfessionalService`, no `BreadcrumbList` anywhere.
- **Impact:** For a personal-brand portfolio + local-services business, `Person` (with `sameAs` ‚Üí GitHub/LinkedIn/X) and `WebSite` are the highest-value schema: they build the knowledge-graph entity and disambiguate brand queries. `services/page.tsx` sells to local businesses ‚Äî `Service`/`ProfessionalService` with `areaServed` is a natural fit.
- **Fix:** Emit in the root locale layout (or `(site)/layout.tsx`):
  ```ts
  { "@context": "https://schema.org", "@type": "Person", name: "Jose ...",
    url: base, jobTitle: "Full-stack developer", sameAs: ["https://github.com/...", "https://www.linkedin.com/in/..."] }
  { "@type": "WebSite", name: "JoseCoded", url: base, inLanguage: ["es", "en"] }
  ```
  Add `BreadcrumbList` to blog articles (Home ‚Üí Blog ‚Üí Post) and forum threads. Note: `FAQPage` rich results are restricted by Google to well-known authoritative sites since 2023 ‚Äî keep the markup (harmless, still parsed) but don't expect the SERP widget.

### M6. Sitemap gaps: fake lastmod, no hreflang alternates, forum content absent
- **Evidence:** `src/app/sitemap.ts` ‚Äî `lastModified: new Date()` on every entry (13-17, 28): every generation stamps "now". Blog entries don't use the post's real `updatedAt`. No `alternates.languages` on entries (Next sitemap supports it and emits `xhtml:link`). Forum thematics/threads (public, indexable) are not listed. `encodeURIComponent(slug)` at line 27 is fine.
- **Impact:** Meaningless lastmod teaches Google to ignore it; missing sitemap-level hreflang wastes the bilingual signal; forum discovery relies purely on internal links.
- **Fix:** Use the post's `updatedAt` for blog `lastModified`; add `alternates: { languages: { es: `${base}/es/...`, en: `${base}/en/...` } }` per entry; append forum thematic + entry URLs from `forumFetchThematics`/`forumFetchEntries` (guarded by `forumIsApiConfigured()` like the Supabase guard at line 21).

### M7. Weak descriptions on blog/foro; blog articles fall back to title as description
- **Evidence:** `screens.blog.metadataDescription` = "Art√≠culos y notas t√©cnicas." (27 chars) / "Articles and technical notes." (29). `screens.forum.metadataDescription` = 39/47 chars. `blog/[slug]/page.tsx:40` ‚Äî `post.excerpt?.trim() || post.title`: articles without excerpt duplicate title into description.
- **Impact:** Short/duplicate descriptions ‚Üí Google rewrites snippets; lost CTR control on exactly the pages meant to earn search traffic.
- **Fix:** Expand both literals to ~140-160 chars with keywords ("desarrollo web, Next.js, apps m√≥viles..."). For articles, fall back to the first ~155 chars of `bodyMd` (stripped of markdown) instead of the title. Make excerpt required in the blog admin flow.

### M8. LCP/image concerns: external Google-hosted case-study image; hero has no image but check `priority` on first visible `next/image`
- **Evidence:** `bento-section.tsx:15-16` hardcodes a `lh3.googleusercontent.com/aida-public/...` URL (a transient AI-generated-image CDN link) rendered via `next/image fill` (lines 100-106) ‚Äî allowed by `next.config.ts:10-18`. Note: **no `images.unoptimized` flag exists in the repo** ‚Äî the "images are now unoptimized for Workers" premise is not yet in code. Plain `next/image` optimization does **not** work on Cloudflare Workers without a custom loader (e.g. Cloudflare Images loader) or `unoptimized: true`; as written, `/_next/image` requests will 404 or fail on Workers.
- **Impact:** (a) The aida-public URL can expire at any time ‚Üí broken image on the homepage case study; (b) third-party image host adds a connection for a mid-page LCP candidate; (c) on Workers, image optimization will break at runtime unless a loader/unoptimized is configured ‚Äî this is a deploy blocker to verify, not just SEO.
- **Fix:** Self-host the case image under `public/` (also gives it a stable URL for OG reuse). For the Workers migration, set either `images: { unoptimized: true }` (fine at this site's scale; serve pre-sized assets) or a Cloudflare image loader. If `unoptimized: true` lands, pre-compress assets (AVIF/WebP, correct dimensions) since srcset generation disappears ‚Äî the SEO impact of `unoptimized` is only indirect (LCP), and with 1-2 images per page it is acceptable.

## LOW severity

### L1. Blog `not-found.tsx` is hard-coded to the default locale
`(site)/blog/not-found.tsx:7` ‚Äî acknowledged in a comment. English visitors hitting a dead article see Spanish. Returns a real 404 status (good). Fix later with a client-side locale sniff from `document.documentElement.lang` or a cookie; not worth complexity now.

### L2. No app-root `not-found.tsx`
Unknown routes under a valid locale (e.g. `/es/nope`) get Next's unstyled default 404. Correct status code, zero brand. Add `src/app/not-found.tsx` (and per-locale copy once M3's `dynamicParams=false` is in, since unknown locales will also land there).

### L3. `keywords` meta on blog articles
`blog/[slug]/page.tsx:44` ‚Äî ignored by Google since 2009. Harmless; keep or drop.

### L4. og:locale / og:locale:alternate / og:url not set
Next derives `og:url` from canonical+metadataBase (fixed by H2/H3). `openGraph.locale: locale === "es" ? "es_ES" : "en_US"` and `alternateLocale` are one-liners in the root layout ‚Äî nice-to-have for Facebook/LinkedIn parsing.

### L5. Empty leftover directories `src/app/stack/` and `src/app/demo/android/`
No route files, so no SEO effect ‚Äî just repo noise. Delete with M2.

### L6. Blog is ISR (revalidate 120), forum is force-dynamic ‚Äî both fine
`blog/[slug]/page.tsx:14` `export const revalidate = 120` + `generateStaticParams` = ISR, not SSR (the audit premise "blog now SSR" doesn't match the code on this branch). Either way Googlebot sees full HTML; ISR is actually preferable (fast TTFB, stable HTML). The forum's `force-dynamic` (`foro/layout.tsx:24`) is fine for SEO as long as Workers TTFB stays reasonable; watch p75 TTFB after deploy since every forum page now fans out to thematics+popular+entries per request (`foro/layout.tsx:49-62`) ‚Äî that fetch fan-out on every crawler hit is a crawl-speed risk, consider short-TTL caching.

## Already good

- 20+ private pages (admin subpages, area-clientes, closer, staff, auth, perfil) consistently carry `robots: { index: false, follow: false }` ‚Äî the discipline exists, only 2 pages + the layout-level safety net are missing.
- Sitemap correctly excludes all private routes and `perfil`, covers both locales, includes blog slugs dynamically with `encodeURIComponent`, and degrades gracefully when Supabase is absent (`sitemap.ts:21`).
- Blog article `generateMetadata` returns `robots: { index: false, follow: true }` for missing posts (`blog/[slug]/page.tsx:37`) ‚Äî correct soft-404 hygiene, paired with a real `notFound()` in the page body.
- Per-locale `<html lang>` (`[locale]/layout.tsx:47`), skip-link, one `h1` per page across home/sobre-mi/services/blog/foro-thread (verified by grep), semantic `<time dateTime>`, `article`/`nav`/`header` usage in blog.
- `next/font/google` for all three families (`dash-fonts.ts`) ‚Äî self-hosted, no render-blocking font CSS, no CLS-prone FOIT.
- `og:type: "article"` + `publishedTime`/`modifiedTime` on blog posts; `type: "website"` elsewhere; `summary_large_image` chosen (just needs the image, H6).
- Locale-scoped `/es|en/stack` ‚Üí `/sobre-mi` 301 is correct; trailing-slash default (none) is consistent everywhere including sitemap.
- FAQPage JSON-LD on home is well-formed and localized; Article JSON-LD is per-locale via `inLanguage`.
- Blog ISR (120s) + `generateStaticParams` is the right rendering mode for SEO; theme script is inline and tiny (no FOUC, no render-blocking external JS).
